# MVP開発プラン

作成日: 2026-07-16
ステータス: ドラフト（統括者レビュー待ち）

このドキュメントは、README「主な機能(MVP)」と CLAUDE.md §0（North Star）・§3（中立性ガードレール）を
実装可能なマイルストーンに落とし込んだ開発プランである。
技術スタック・新規依存は本プランでは**提案**にとどめ、統括者の承認をもって確定する（CLAUDE.md §10）。

---

## 1. MVPの目的と検証したい仮説

North Star:「岩盤規制によるQoL低下の事実を生活者が体感できる形で可視化し、世論形成につなげる」

MVPで検証する仮説は以下の2つに絞る。

| # | 仮説 | 検証方法 |
|---|---|---|
| H1 | 実在のパブコメ結果（意見・回答ペア）を、価値判断を混入させずに構造的指標へ機械変換できる | 分類ロジックのコード公開＋原文との突合レビューで確認 |
| H2 | 構造的指標の可視化だけで「物語の種」（第三者が引用・発信したくなる事実）になる | 公開後のSNS言及・引用の有無（レベル3指標として測定のみ、機能要件にはしない） |

**MVPは「単一案件の可視化パイプラインを一気通貫で公開する」ことをゴールとする。**
提出機能・複数省庁比較は含まない（§5 スコープ外を参照）。

## 2. 対象データ（事前検証の結果）

Phase 1 の最初の対象は README どおり「交通政策基本計画パブコメ結果」とする。
2026-07-16 時点で以下を確認済み。確定した取得元URLは M1 で `docs/SOURCES.md` に正式記録する。

### 確認済みの事実
- 第3次交通政策基本計画は 2026-01-16 に閣議決定済み
  （国交省報道発表: https://www.mlit.go.jp/report/press/sogo12_hh_000496.html ）
- 素案への意見募集は e-Gov 上に案件として存在（案件ID: 155250107、意見募集開始 2025-10-31）
- 参考: 第2次計画（2021年）のパブコメは 64者145件 の意見があり、結果はPDFで公表された実績がある

### 技術的制約（取得戦略に影響）
- e-Gov パブリックコメントの現行サイトは JavaScript SPA であり、単純な HTTP GET では
  案件データを取得できない（エラーページの静的シェルのみ返る）。旧servlet形式URLも同様。
- 一般的なフェッチャーのUser-Agentは 403 で拒否される。

### 未検証（M1冒頭で必ず確認）
- 第3次計画の「結果の公示」ページの正確なURLと、意見・回答一覧の**ファイル形式**（PDF想定・未確認）
- 国交省サイト側（報道発表添付資料）に同内容のPDFがあるか（e-Govを介さない取得経路の有無）

### 取得方針（上記制約からの帰結）
MVPは対象が1案件のため、**スクレイパーは作らない**（Simplicity First・CLAUDE.md §11 の過剰汎化回避）。
- 結果公示PDFを**手動ダウンロード**し、取得元URL・取得日時・ファイルハッシュを `docs/SOURCES.md` に記録する
- 原文PDFはリポジトリに改変せず保存する（§7: 原文の改変・恣意的削除の禁止）
- e-Gov API / 自動取得の調査は Post-MVP の課題とする

## 3. マイルストーン

### M0: 土台整備（実装なし・ドキュメントのみ）
| 成果物 | 内容 |
|---|---|
| `docs/SOURCES.md` | データ取得記録の様式を定義（取得元URL・取得日・ハッシュ・原文の保存場所） |
| `LICENSE` | MIT（README記載済みの方針） |
| `docs/GOVERNANCE.md` | 骨子のみ: 運営体制・資金源の開示方針、中立性の限界の開示（§3.4） |
| 技術スタック確定 | §6 の提案を統括者が承認・修正（**M1以降の着手条件**） |

成功条件: 統括者が技術スタックを承認し、SOURCES.mdの様式が確定している。

### M1: データ取得・正規化
| 成果物 | 内容 |
|---|---|
| 原文データ | 結果公示PDF（改変なし）＋ SOURCES.md への記録 |
| 抽出スクリプト | PDF → 意見・回答ペアの構造化データ（JSON）。1行=1意見（意見要旨・行政回答・付帯情報） |
| 突合レポート | 抽出結果と原文PDFの目視突合（全件 or 件数が多い場合は統括者と合意したサンプル数） |

成功条件（検証可能）:
- 抽出された意見・回答ペア数が原文の記載件数と一致する
- ランダム抽出したNペアの本文が原文と文字単位で一致する（Nは突合レポートで明記）
- 抽出できなかった箇所は「その他」に押し込まず、欠損として明示する（§3: 実質的な歪曲の回避）

### M2: 構造的指標の実装（§3.1 の範囲のみ）
実装する指標（すべて README 記載のMVP機能に対応）:

| 指標 | 実装内容 | §3.1 根拠 |
|---|---|---|
| 文字数 | 回答本文の文字数集計 | 「不採用理由の文字数の集計」 |
| 定型句検出 | 「ご意見として承りました」等の定型句リストとの照合。**リスト自体をコードで公開** | 「定型句の検出」 |
| 使い回し検出 | 同一・類似文言が複数の意見への回答に使われている割合 | 「同一文言の使い回し検出」 |
| 具体的参照の有無 | 条文番号・統計データ等への言及の機械的検出 | 「具体的参照の有無の機械的検出」 |
| キーワード分類 | 「法的根拠」「財政的制約」「技術的困難」等への分類。ロジック公開 | 「キーワードベースの分類」 |

実装ルール:
- 各ロジックの冒頭コメントに「何を中立的事実として扱い、何を評価しないか」を明記する（CLAUDE.md §6）
- テストは `test_<対象>_<条件>` 形式で正常系・異常系・境界値を網羅（§8）。外部アクセスは発生しない設計
  （M1で取得済みのローカルデータのみを入力とする）
- 類似判定のしきい値・定型句リストは設定として切り出し、変更時に分類結果の差分を提示する（§8）

成功条件: 全指標が取得済みデータに対して動作し、テストが通り、分類結果の差分レポートが出力できる。

### M3: 可視化・公開
| 成果物 | 内容 |
|---|---|
| 案件ページ | 意見・回答ペアの横並び表示＋各指標の表示（**評価コメントなし**） |
| 集計ページ | 案件内の指標集計（定型句使用率、文字数分布、分類別件数、使い回し検出結果） |
| 限界の開示 | 「この指標が測っていること・測っていないこと」の説明文を各指標に併記 |

公開形態は静的サイト（§6参照）。公開前に §7 の中立性ガードレールチェック表を統括者に提示する。

成功条件: 第三者が「元PDF → 構造化データ → 指標 → 表示」の全過程をリポジトリ内で追跡できる。

### M4（Post-MVP・本プランのスコープ外）
- 複数案件・複数省庁への拡張と「メニュー化ランキング」（不採用率・回答日数・定型句使用率の並列提示）
  ※ ランキングはREADMEのMVP機能だが、**比較対象が複数ないと成立しない**ため、
    2案件目（住宅・空き家関連 = Phase 2、または国交省の別案件）のデータ化後に着手する
- e-Gov の API / 自動取得手段の調査（利用規約・robots.txt の確認を含む）
- パブコメ提出導線（レベル1 KPI「提出完了数」はこの段階から計測可能になる）

## 4. KPIとの対応

| マイルストーン | 対応するKPI（CLAUDE.md §4） |
|---|---|
| M1–M2 | レベル2: 分類の網羅率（欠損・「その他」率を含めて公開） |
| M3 | レベル2: 説明責任データの公開状況 |
| M4以降 | レベル1: 提出完了数等（MVPでは測定対象なし） |

レベル3（メディア掲載・SNS言及）は測定のみ行い、機能要件・成功条件には使わない。

## 5. スコープ外（MVPでやらないこと）

- 総合スコア・総合ランキング（§3.2 恒久的禁止）
- AI/LLMによる回答の質的評価、評価ラベル（§3.2 恒久的禁止）
- ユーザー投票・評価機能（実装する場合は組織票対策のレビューが先。MVPでは対象外）
- パブコメ提出機能・ユーザー登録・通知（M4以降）
- e-Gov汎用スクレイパー（1案件に対して過剰。Post-MVPで調査から始める）
- DB・サーバーサイド（静的サイトで足りる規模。§1 Simplicity First）

## 6. 技術スタック（提案・統括者承認待ち）

新規依存の追加は統括者承認が必要（CLAUDE.md §10）なため、以下は**提案**である。

| 領域 | 提案 | 理由 |
|---|---|---|
| データ処理・分類 | Python 3.12 + 標準ライブラリ中心 | 分類ロジックの可読性・公開性を優先。個人開発の学習コストも低い |
| PDF抽出 | pdfplumber（依存追加・要承認） | 表形式PDF抽出の定番。代替: pypdf（表に弱い）、手動転記（件数次第では現実解） |
| テスト | pytest（依存追加・要承認） | §8 のテスト規約を素直に書ける |
| 可視化・公開 | 静的HTML/JSON生成 + GitHub Pages | サーバー不要・無料・リポジトリと公開物が同一で追跡可能性が高い |
| CI | GitHub Actions（テスト実行のみ） | 分類ロジック変更時の差分検出を自動化 |

判断が必要な点:
1. pdfplumber / pypdf / 手動転記のどれにするか（結果PDFの実物確認後 = M1冒頭に最終判断）
2. GitHub Pages でよいか（独自ドメインの要否は M3 で判断）

## 7. 中立性ガードレールチェック（§3 照合・§9 Output Rules）

| MVP機能 | §3判定 | 備考 |
|---|---|---|
| 文字数・日数の集計 | ✅ 3.1で明示的に許可 | — |
| 定型句・使い回し検出 | ✅ 3.1で明示的に許可 | 定型句リスト・しきい値をコードで公開 |
| 具体的参照の有無 | ✅ 3.1で明示的に許可 | キーワード検出のみ。参照の「適切さ」は評価しない |
| キーワード分類 | ✅ 3.1で明示的に許可 | ロジック公開。カテゴリ設計は複数案を列挙して統括者確認（§2） |
| 意見・回答の横並び表示 | ✅ 3.1で明示的に許可 | 評価コメントは付けない |
| 集計ページ | ⚠️ 要注意 | 単一案件内の集計であり省庁間比較ではないが、指標の並べ方・既定の並び順が   暗黙の順位づけにならないよう、M3のレビューで確認する |
| 総合スコア・質的評価 | ❌ 実装しない | §3.2 |

残る限界（開示対象）:
- 定型句リストの選定・類似判定のしきい値・分類カテゴリの設計には運営の判断が入る。
  → すべてコードで公開し、`docs/GOVERNANCE.md` と各指標の説明文で限界として明示する。

## 8. リスクと対策

| リスク | 対策 |
|---|---|
| 結果PDFのレイアウトが複雑で機械抽出できない | 件数が少なければ手動転記に切替（原文PDFとの突合で正確性を担保）。M1冒頭に実物で判断 |
| 結果公示に意見・回答の対応表が含まれない（要旨のみ等） | その時点で統括者に報告し、対象案件の変更を協議（推測で補完しない） |
| 定型句リストが「恣意的」と批判される | リスト・追加基準を公開し、Issue/PRで修正提案を受け付ける（README記載の方針） |
| 単一案件のデータで一般化した主張に見えてしまう | 全ページに対象案件・データ範囲を明記。「国交省全体」等への言い換えをしない |
