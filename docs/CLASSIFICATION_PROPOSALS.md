# キーワード分類のカテゴリ設計案（統括者確認用）

作成日: 2026-07-16
ステータス: **確定 — 2026-07-16 統括者承認: 案A＋案Bの併用**

## 決定記録

- 案A: 実装済み（抽出データの `group` フィールド。行政自身のⅠ/Ⅱ区分をそのまま保持）
- 案B: `analysis/metrics.py` の `RESPONSE_CATEGORIES` として実装（マルチラベル方式）
- 案C: 本案件には適用せず、規制改正系案件の取得時に再検討
- S-001 での初回分類結果（以後の変更時はこの結果との差分を提示する）:
  記述追加・修正 3件 / 既記載 25件 / 素案のとおり 5件 / 検討継続 1件 /
  既存取組 3件 / その他 4件（その他率 10.5% — Ⅱ-4, Ⅱ-5, Ⅱ-9, Ⅱ-17）

分類カテゴリの設計は、構造的指標の中で最も運営の判断が混入しやすい箇所である
（どのカテゴリを作るか・「その他」に何を落とすか。CLAUDE.md §3.1）。
そのため、実データ（S-001、38件）の観察に基づく複数案を列挙し、選択を統括者に委ねる。

## 実データの観察（設計の前提）

S-001 の回答38件に「採用/不採用」の明示ラベルは**存在しない**。
回答文は以下の構文パターンに大別できることを確認した（機械的な文字列観察）:

| 観察されたパターン | 例 | 件数の目安 |
|---|---|---|
| 「…記述を追加しました」 | Ⅰ群（3件） | 3 |
| 「…記述しているところであり、…適切に対処して参ります」 | Ⅱ-6, Ⅱ-10〜16 ほか | 20超 |
| 「素案のとおりとして…」「素案のままとし…」 | Ⅱ-1, Ⅱ-2, Ⅱ-15, Ⅱ-32, Ⅱ-35 | 5 |
| 「…取り組んでおります」（既存取組の説明のみ） | Ⅱ-7, Ⅱ-19 | 2 |
| 「引き続き検討を進めてまいります」 | Ⅱ-8 | 1 |
| 上記いずれにも該当しない | Ⅱ-4/5/9（ライドシェア）ほか | 数件 |

また、当初想定していた「法的根拠」「財政的制約」「技術的困難」という**不採用理由の
種別語彙は、本案件の回答にはほぼ出現しない**。これは本案件が「計画策定」パブコメであり、
「規制改正」パブコメとは回答の語彙構造が異なるためと考えられる。

## 案A: 元資料のグループ構造をそのまま分類に使う

- カテゴリ: 「Ⅰ．当該意見を踏まえて内容を修正するもの」「Ⅱ．その他の主な意見」（原文ママ）
- 分類主体: 国交省自身（本プロジェクトは分類判断を一切しない）
- メリット: **価値判断の混入がゼロ**（行政自身の区分をそのまま表示するだけ）。実装済み
- デメリット: 粒度が粗い（2値）。省庁がこの区分を設けない案件では使えない

## 案B: 回答文の構文パターン分類（キーワードベース）

- カテゴリ案（名称は事実記述に徹し、評価語を使わない）:
  1. 「記述を追加・修正した」旨を含む回答
  2. 「既に記述している」旨を含む回答
  3. 「素案のとおりとする」旨を含む回答
  4. 「検討を継続する」旨を含む回答
  5. 「既存の取組を説明する」回答
  6. その他（上記キーワードに該当しない）
- 分類主体: 運営（ただしキーワードルールはコードで全公開、変更はPR経由＋差分提示）
- メリット: 粒度が細かく、案件間比較（M4のメニュー化ランキング）の土台になる
- デメリット: カテゴリ境界の設計に運営判断が入る。「その他」の中身の監視が必要
  （「その他」率が高い案件では分類の意味が薄れるため、率を常に併記する）

## 案C: 不採用理由の種別分類（法的根拠/財政的制約/技術的困難）

- 当初プラン（CLAUDE.md §3.1 の例示）に沿った案
- **本案件には不適合**: 該当語彙がほぼ出現しないため、ほぼ全件が「その他」になる
- 規制改正系パブコメ（Phase 1 本命のライドシェア規制の個別案件等）では有効な可能性が
  あるため、**廃案にせず将来案件用に保留**することを推奨

---

## 推奨

**案A（即時採用）＋案B（統括者が文言を確認の上で採用）の併用**を推奨する。

- 案Aは実装済み（抽出データの `group` フィールド）であり、追加の判断が不要
- 案Bはランキングのメニュー化（M4）に必要な粒度を持つが、カテゴリ名・キーワードの
  確定に統括者確認が必要
- 案Cは本案件では適用せず、規制改正系案件の取得時に再検討する
